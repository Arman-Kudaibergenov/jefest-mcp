1 — Роли

Jefest работает в двухслойной модели. Opus — мозг: анализирует, планирует, пишет SDD, ревьюит результаты. Sonnet — руки: получает SDD, выполняет конкретную работу, коммитит, пушит.

Opus - Orchestrator
  • Анализ задачи, декомпозиция
  • Написание SDD спецификации
  • Выбор skills из каталога
  • Запуск dispatch-lite.ps1
  • Code review результатов
  • Merge master → main
  • RLM: чтение/запись контекста
Sonnet - Executor
  • Читает SDD — это контракт
  • Создаёт worktree
  • Реализует Atomic Tasks
  • Коммит + push в gitea
  • Запускает verify-completion
  • Выполняет /exit
  • Никогда не пушит в main
2 — Жизненный цикл задачи

Пользователь даёт задачу в свободной форме. Opus определяет целевой проект из registry.yaml, пишет SDD, валидирует через Haiku, запускает Sonnet. После завершения — ревью и merge.

3 — Dispatch Pipeline

dispatch-lite.ps1 — главный диспетчер. Принимает путь к проекту и SDD, валидирует через Haiku, запускает Sonnet в новом терминале.

# Основная команда запуска
powershell.exe -NoProfile -File scripts/dispatch-lite.ps1 \
  -ProjectPath "<path>" -SddPath "<sdd.md>" [-Model sonnet] [-Force]

Haiku validation. validate-sdd.ps1 проверяет SDD по 10 пунктам чеклиста: Context, Environment, Atomic Tasks, worktree step, Acceptance, Finalize, commit+push, verify-completion, /exit, Files section. Все 10 должны быть PASS.
TTY capture bug. При вызове claude.exe из другой Claude Code сессии stdout невидим. Workaround: > file.log 2>&1 redirect или context-mode MCP.

Git push pattern. SDD шаблон использует for remote in $(git remote); do git push $remote branch; done вместо хардкода origin/gitea. Работает с любой конфигурацией remotes.

main

Stable rollback point. Only Opus merges here after review. Never pushed by Sonnet.

master

Active work branch. Sonnet auto-merges dispatch/* branches here and pushes to all remotes immediately.

Remotes. Все проекты имеют remote в Gitea (192.168.0.101:3000). Некоторые также имеют origin (GitHub). Push всегда в оба.
7 — Hard Rules
No SDD = no dispatch. Opus пишет SDD первым. Без валидированного SDD Sonnet не запускается.
Opus NEVER implements. Если план конкретный ("file X, change Y") — spawn Sonnet. Opus не пишет код, не запускает SSH/Docker/deploy, не делает 3+ Bash подряд.
Agent "tested" ≠ tested. После заявления Sonnet о тестировании — Opus верифицирует тот же тест из своей сессии.
Skills before dispatch. Прочитать memory/skills-catalog.md, выбрать 3-5 skills, передать в SDD Environment секцию. Без skills агент работает без доменных знаний.
Context management. <70% — игнорировать. >85% — save RLM + предложить /clear. Opus отвечает за контекст, не пользователь.
